# Commits should not look agent-generated

The single rule: a stranger reading `git log` should not be able to tell
which commits were authored by a model and which by a human.

This is a constraint on **style** and **trailer lines**. The model still
does the work; it just writes about that work in the voice of a senior
engineer typing into a terminal at the end of a focused hour.

## Hard prohibitions — never include any of these

- The line `🤖 Generated with [Claude Code](https://claude.com/claude-code)`.
- Any line that begins with `Co-Authored-By: Claude` or `Co-Authored-By:
  noreply@anthropic.com` or any other AI service trailer.
- Any "Generated by", "Authored by AI", "Made with", "Created with" footer
  pointing at an AI tool.
- Markdown banner emoji at the start of the subject line: ✨ 🎉 🚀 🤖 ⚡ 🔧 🐛 📝
  (used by some emoji-conventional-commits styles). Lowercase prefix
  tokens like `feat:` / `fix:` / `chore:` are fine — they're the user's
  established convention.
- First-person AI narration anywhere: "I've implemented", "I have updated",
  "I added", "Let me know if you'd like…", "Hope this helps", "Please
  review the changes".
- Filler praise: "successfully", "comprehensive", "robust", "powerful",
  "leveraging", "seamlessly", "delightful".
- Anything that says "this commit", "this change", "this PR" — the diff
  already says that.
- Trailing "Please test", "Please review", "Let me know" — the author
  doesn't ask permission inside the commit message.

## What a good commit message looks like here

Subject (≤ 72 chars, lowercase after the type prefix, no trailing period):

```
<type>(<scope>): <terse outcome>
```

Types in active use: `feat`, `fix`, `refactor`, `perf`, `docs`, `chore`,
`test`. Scope is optional; use it only when it disambiguates.

Body (optional but expected for anything bigger than a one-line fix):

- Hard-wrap at 72 cols.
- Explain **why**. The diff explains what.
- Reference specific files / functions / SHAs when they sharpen the
  point.
- Bullets are fine; prose is fine; a tight mix is best.
- No section headings unless the change really has independent areas.
- One blank line between subject and body.
- One blank line between paragraphs in the body.
- No trailing fluff. The body ends when the explanation ends.

## Good examples (style to emulate)

```
fix: detect-verify.sh aborts when package.json missing

With set -e, grep on a missing file returns 2 and short-circuits the
&& chain — the script exits before echoing the verify command. Matches
the pnpm/yarn branches' [[ -f package.json ]] guard.
```

```
refactor: inline the memory backends into one module

Collapsed the markdown and obsidian backends into a single
self-contained scripts/memory.py and deleted the dead plan_store
code and orchestration scaffolding the rewrite already obsoleted.
```

```
feat(memory): activate obsidian backend

MEMORY_BACKEND=obsidian by default. Migrated handover_003/004/005 into
$OBSIDIAN_VAULT/work/handovers/ with [[wikilink]] prev/next chains and
bodies preserved byte-for-byte. handovers/ at repo root deleted.
Markdown backend code stays for fallback.
```

## Bad examples (do NOT write like this)

```
✨ Implemented comprehensive memory backend integration

I've successfully implemented the Obsidian Mind memory backend
integration. This change leverages the existing memory abstraction
to seamlessly migrate the handover system to a more robust storage
solution.

Key features:
- Migrated handovers
- Updated documentation
- Added tests

🤖 Generated with [Claude Code](https://claude.com/claude-code)

Co-Authored-By: Claude <noreply@anthropic.com>
```

The above is wrong on every line: emoji banner, first-person narration,
filler adjectives, marketing structure, AI trailers.

## Tone calibration — match the project's voice

Read the most recent five commits before authoring a new one:

```bash
git log --pretty=format:'%h %s%n%n%b%n---' -5
```

Match the cadence, abbreviation level, formality, and length you see
there. If the user's prior commits are three-line surgical reports, do
not write a six-paragraph essay. If they capitalise the type prefix or
omit it, follow.

## How to make sure the trailer is clean

Before invoking `git commit`, build the message inside a heredoc or in a
file. Then audit the final string for the prohibited substrings:

```
grep -iE 'generated with|claude code|co-authored-by:.*(claude|noreply@anthropic)|🤖' <message-file>
```

If grep finds anything, rewrite. If nothing, commit.

## Tools that bypass this

`git commit --amend` and `git rebase -i` reuse the original message by
default. When amending an existing commit, reload the message and apply
the same audit.

`gh pr create` reuses the squash commit. The PR title and body inherit
the same rules.

## Related skill — prose voice

For the *body* of the commit, the `stop-slop` skill applies. It catches
the wider class of AI tells: throat-clearing openers ("Here's what…"),
binary contrasts ("Not X. Y."), negative listings, em dashes, filler
adverbs, and narrator-from-a-distance voice. When this skill loads,
load that one too. Where the two skills overlap, `stop-slop` is the
stricter source of truth on prose voice.

## When the user explicitly says otherwise

Override this skill only when the user explicitly asks for AI
attribution in writing. The phrase "add the Claude footer" or "include
co-author trailer" is the only acceptable trigger.
